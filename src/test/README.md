# テストを書くルール

## 定数

- 複数のテストで使い回す値は、テスト内に直接書かずに `src/test/const.py` に定義する。
- 外部サービス、時刻、UUID、ランダム値など、実行ごとに結果が変わる依存は固定化し、テスト結果が不安定にならないようにする。
- UUID やランダム値は、生成されること自体を確認するテストを除き、`src/test/const.py` に定義した固定値に置き換える。

## fixture

- fixture は `conftest.py` に定義する。
- fixture は、基本的に利用範囲が最も小さくなる階層の `conftest.py` に配置する。
- 他の範囲でも利用することが想定される場合、または実際に他の範囲から利用する必要が出た場合は、共通して参照できる上位階層の `conftest.py` に移動する。

## テストの書き方

- テスト関数名は `test_<関数名>_success_<詳細>` または `test_<関数名>_failure_<詳細>` の形式で記述する。
- 正常系のテストは `success`、異常系のテストは `failure` を使用する。
- `<詳細>` には、テスト条件や期待する振る舞いが分かる内容を記述する。
- テスト本文では `# Arrange`、`# Act`、`# Assert` のコメントを使用して、準備、実行、検証を分ける。
- 例外を期待するテストでは `pytest.raises` を使用し、必要に応じて例外内容も検証する。
- 1 つのテスト関数では、検証する責務や観点を 1 つに絞る。
- テスト関数は、責務が単一である範囲で可能な限り少なくする。
- 同じ責務・同じ期待結果の入力違いは、`pytest.mark.parametrize` などを使用して 1 つのテスト関数にまとめる。
- ただし、責務や検証観点が異なる場合は、無理にまとめず別のテスト関数に分ける。
- 例: `set_title` に不正な文字列や空文字を渡すケースは、個別のテスト関数に分けず、`invalid_value` のテストとしてまとめる。
- テスト対象のクラスが Repository などの依存オブジェクトを必要とする場合は、必要に応じて Dummy クラスを作成し、差し替えてテストする。
- Dummy Repository や Dummy クラスは、テスト対象の振る舞いを確認するために必要な最小限の実装にする。

## サンプル

```python
import pytest


@pytest.mark.parametrize(
    "invalid_title",
    [
        "",
        "a" * 101
    ],
    ids=["empty", "too_long"],
)
def test_set_title_failure_invalid_value(textbook, invalid_title):
    # Arrange
    expected_error = ValueError

    # Act
    def act():
        textbook.set_title(invalid_title)

    # Assert
    with pytest.raises(expected_error):
        act()
```

## テストパターン

- テストケースを作成する際は、正常系、異常系、境界値、必須項目、フォーマット不正、状態変化、不変条件、重複・一意性、依存処理、副作用、冪等性、例外内容、ランダム値・時刻、回帰テストの観点を考慮する。
- すべての観点を必ず網羅するのではなく、テスト対象の責務や振る舞いに応じて必要なパターンを選択する。

## 注意事項

- テストは実行順序に依存せず、単体で実行しても成功するようにする。
- 更新系のテストでは、変更対象の値だけでなく、変更されない値が維持されていることも確認する。
